# scikit-agent Project Governance

The scikit-agent project is based on the recognition that the design of our
technical systems is the result of many human choices. We aim to keep our
governance lightweight, merit-based, and transparent. This document details the
bylaws of the scikit-agent community.

## The Project

The scikit-agent project (The Project) is an open source software project. The
goal of The Project is to develop open source software for research and
education. The Software developed by The Project is released under the MIT
license, developed openly and hosted on public GitHub repositories.

The Project is developed by a team of distributed developers, called
Contributors. Contributors are individuals who have contributed code,
documentation, designs or other work to one or more Project repositories. Anyone
can be a Contributor, affiliated with any legal entity or none. Contributors
participate by submitting, reviewing, and discussing GitHub Pull Requests and
Issues and by participating in open, public Project discussions. The foundation
of Project participation is openness and transparency.

The current Contributors to the main scikit-agent repository are listed here:

https://github.com/scikit-agent/scikit-agent/graphs/contributors

The Project Community consists of all Contributors and Users of the Project. We
strive to keep the barrier between Contributors and Users as low as possible.

All members of the Project Community are expected to abide by and maintain the
Code of Conduct, in order to make participation in our project and community a
harassment-free experience for everyone.

## Governance

The foundations of Project governance are:

- Openness & Transparency
- Active Contribution
- Institutional Neutrality

Decisions are made by **rough consensus** of the Core Developers, with input
from the Community. Rough consensus means that a decision can proceed once it
has support among the active Core Developers and any sustained objections have
been heard and addressed. Most decisions are made informally in the course of
reviewing Pull Requests and Issues and never require a formal vote.

### Core Developers

Core Developers are Contributors whose active and consistent contributions have
been recognized by the granting of commit rights to the Project repositories.

A Contributor becomes a Core Developer by consensus of the current active Core
Developers. Any active Core Developer may nominate a Contributor; the nomination
is approved if no active Core Developer objects.

The current Core Developers, updated by the Steering Committee after each
induction, are:

- Sebastian Benthall
- Alan Lujan

Core Developers are expected to:

- Make useful contributions — commits or reviews — at least once in any 9-month
  period. A Core Developer who does not becomes Inactive and holds the same
  privileges as a Contributor until they contribute again.
- Review contributions from Core Developers and Contributors, either giving
  concrete guidance on how to improve the change or merging it and notifying the
  submitter.

Core Developers may "self-review" and commit directly, but should reserve this
for changes they judge to be extremely minor.

### Resolving conflicts

When rough consensus cannot be reached on a significant or far-reaching change,
any Core Developer may refer the decision to the Steering Committee, which
resolves it by majority vote. The Steering Committee should seek consensus first
and vote only when necessary.

## Steering Committee

The Steering Committee is the Project's board for decisions that consensus
cannot settle, including changes to these bylaws. It is made up of the top 5
Core Developers over the past 365 days by number of commits. If fewer than five
active Core Developers are available, the Contributors with the most commits in
the past 365 days fill the remaining seats.

Changes to these bylaws should be made by consensus if at all possible, and
otherwise by majority vote of the Steering Committee.

In the event of a tie, the status quo prevails: the proposed change does not
take effect.

## Private communications

Unless specifically required, all governance-related discussions and activities
are public and conducted in collaboration with the Project Community. Private
communications are used sparingly and only when a matter requires privacy. When
they are needed, the Core Developers will do their best to summarize them to the
Community afterward, eliding personal, private, or sensitive information that
should not be posted publicly.
