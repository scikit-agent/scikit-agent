## D-2 convergence investigation log (2026-05-27)

Sequence of interventions tried, what each surfaced, and what it implies. Keeps
the diagnostic trail visible for future hyperparameter tuning and for the
educational write-up.

### Step 1 — baseline (200 episodes × 100 steps, defaults)

Symptom: actor saturated at `c = m`, MAPE 1.59 (159% off). Reward never improved
from the "consume everything" steady state.

### Step 2 — warmup phase (`warmup_episodes=20`)

Implemented `DDPG.get_random_decision_rule()` and a warmup loop that fills the
replay buffer with uniform-random actions before actor updates begin. Critic
gains exposure to diverse `(a, c)` pairs. Significant improvement reported, but
actor still far from analytical solution.

### Step 3 — scale-adaptive OUNoise (`noise_scale_frac=0.3`)

OUNoise now produces noise scaled to the per-state action range,
`noise · (hi − lo) · noise_scale_frac`. Continued exploration after warmup ends.
Improvement observed but plateau remained — learned policy approximated
`c = m − constant` (slope 1, varying offset).

### Step 4 — periodic random rollouts (`random_rollout_every=20`)

Every 20 main episodes, run a random-policy episode (training updates still
applied) to refresh buffer diversity. _Did not_ materially move convergence —
diagnostic at 50 episodes shows actor again saturated at `c = m`, reward stuck
at ≈−99.

### Step 5 — diagnostic interpretation (current state)

Ran `ddpg_diagnostic.py`. Three concrete findings from the Q-value comparison:

- **Critic magnitude is ~half of true Q\*.** Q*critic ≈ −13, Q\* ≈ −24. The
  bootstrap chain has not propagated through enough updates: with γ=0.96 the
  effective horizon is `1/(1−γ) = 25`, and at `tau=0.005` the target tracks ~200
  updates behind, so we need on the order of `25 · 200 = 5000` updates \_just to
  propagate the value chain*. We are exactly at that boundary.
- **Critic is flat in c.** At `a=1`, sliding `c` from 1.22 → 2.03 changes
  Q_critic by 0.23. The true Q surface should change visibly (and curve sharply
  for small c). With `∂Q_critic/∂c ≈ 0` the actor gets no slope signal —
  explains the saturation.
- **Steady-state collapse.** Once the actor outputs `c ≥ m − ε`, the next
  arrival state is `a' ≈ 0`, so `m' = y`, `c' ≈ y`, and the agent sits in the
  `a=0, m=y, c=y` absorbing state for the rest of the episode. Buffer stops
  seeing state diversity even with periodic rollouts unless the rollouts
  dominate the data.

The exploration interventions (steps 2–4) are _necessary_ but not _sufficient_:
until the critic's Q values are calibrated in magnitude and have meaningful
curvature in `c`, the diverse `(a, c)` data going into the buffer can't produce
a useful policy gradient.

### Step 6 — planned: critic-dominated training (in progress)

The next set of interventions push DDPG toward the fitted-Q end of the spectrum
(see section below). Concretely:

- Bump `tau` to 0.01–0.02 so targets propagate faster.
- Run multiple critic gradient steps per env step (TD3-style; needs new param
  `critic_updates_per_step`).
- Increase `random_rollout_every` density (e.g. one random episode per main
  episode, or `random_rollout_episodes=2`).
- Longer training: `num_episodes=2000+` rather than 200.
- Consider critic-only pretraining: train the critic for N gradient steps on the
  warmup buffer _before_ the actor takes a single step.

The goal is to find a parameter regime that hits MAPE < 0.25 convincingly on D-2
and to keep that regime as the test default. The successful parameters then
become the educational reference point — "this is what 'enough' looks like" —
with weaker settings shown in the example notebook to illustrate failure modes.

## Fitted-Q convergence levers (DDPG ↔ fitted-Q spectrum)

DDPG and fitted-Q iteration are two ends of a spectrum, related by _how
on-policy the behavior data is_ and _how much critic training happens per actor
update_. This is the key educational story: dialing certain parameters slides
DDPG toward neighboring algorithms, and each lever has a corresponding effect on
convergence behavior.

| Lever                                | DDPG end                     | Fitted-Q end                             | Mechanism                                                                        |
| ------------------------------------ | ---------------------------- | ---------------------------------------- | -------------------------------------------------------------------------------- |
| Behavior policy                      | actor + small OU noise       | uniform random over action range         | controls coverage; uniform random gives fully off-policy data                    |
| `warmup_episodes`                    | small (0–50)                 | very large (all data, no actor training) | fitted-Q is the limit `warmup → ∞`, then actor trains on fixed buffer            |
| `random_rollout_every` / `_episodes` | 0 / 1                        | every step is random                     | mid-training off-policy injection                                                |
| `noise_scale_frac`                   | 0.1–0.3 (local exploration)  | 1.0+ (covers full range)                 | tradeoff between policy refinement and coverage                                  |
| Critic updates per env step          | 1 (current)                  | many (e.g. 5–10)                         | fitted-Q updates the value estimate to convergence between data collection       |
| `tau`                                | 0.005 (slow target tracking) | 1.0 (target = critic, hard updates)      | hard updates ≈ classical Q-learning; slow updates stabilize DDPG bootstrap       |
| Actor update frequency               | every step                   | rare / disabled                          | TD3 delays actor; fitted-Q derives policy as `argmax_a Q` only after Q converges |
| Reward scaling                       | raw                          | normalized                               | accelerates critic convergence regardless of magnitude                           |

**Limiting behaviors worth documenting in examples:**

- `warmup_episodes = num_episodes, num_episodes = 0` → pure fitted-Q on
  uniform-random data (still trains critic, never updates actor; policy is
  `argmax_a Q(s, a)` over a grid at the end). Not currently supported by
  `ddpg_training_loop` but easy to add.
- `tau = 1.0, critic_updates_per_step = many` → near-classical Q-learning with
  neural function approximation, deterministic policy extracted from the critic.
- Standard DDPG (current defaults) → balance between on-policy refinement and
  off-policy stability.

**Parameters not yet exposed but needed to tell this story:**

- `critic_updates_per_step: int = 1` — how many gradient steps the critic takes
  per env step. Cheap to add to `DDPG.train()` / the loop.
- `actor_update_every: int = 1` — TD3-style delayed actor updates.
- `pretrain_critic_steps: int = 0` — train critic on warmup buffer this many
  gradient steps before the actor is touched at all.
- `reward_scale: float = 1.0` — multiplier on stored rewards.

These should be added with the same "exposed knob, sensible default, documented
limiting case" pattern as `warmup_episodes` and `noise_scale_frac`.

## Broader roadmap: value/Q network API over the three Bellman moments

The `BlockQNet`-vs-`BlockValueNet` inconsistency is a symptom of a deeper design
gap: skagent's value-function classes don't make explicit _which_ Bellman
quantity they represent. Within a single period there are three distinct value
objects, each at a different "moment" relative to shocks and the action:

| Moment                          | Object         | Input              | Bellman relation                            |
| ------------------------------- | -------------- | ------------------ | ------------------------------------------- |
| Before shocks (period boundary) | `V_pre(s_arr)` | arrival states     | `V_pre(s_arr) = E_η[V_post(I(s_arr, η))]`   |
| After shocks, before action     | `V_post(I)`    | info-set           | `V_post(I) = max_c Q(I, c)`                 |
| After shocks and action         | `Q(I, c)`      | info-set + control | `Q(I, c) = u(c, I) + β·V_pre(s_arr'(I, c))` |

These are distinct, not aliases: `V_pre` and `V_post` differ by integration over
shocks, and the `V_post = max_c Q` identity only holds when both functions share
the info-set as input. Picking the wrong one silently breaks the Bellman
recursion (the current `BlockQNet` situation), confounds reviewers, and — most
importantly — has no consistent extension to **partially observed** or
**multi-agent** problems, where the only well-defined decision-time state is the
agent's own info-set.

### Where the current code sits

- `BlockPolicyNet`: input is `iset`. ✓ (decision-time)
- `BlockValueNet`: input is `iset`. → represents `V_post`. ✓
- `BlockQNet`: input is `arrival_states`. → mislabeled; should be `iset` to
  represent `Q(I, c)`.
- `V_pre` is **not represented at all**. It's currently implicit / inlined in
  algorithms that need it (e.g. VBI-style backward induction).

### Proposed API direction

A clean API names each moment explicitly, so the type tells you what Bellman
quantity is being modeled:

```
BlockArrivalValueNet(bp, agent=...)          # V_pre(s_arr) — over arrival states
BlockDecisionValueNet(bp, control_sym=...)   # V_post(I)   — over iset (current BlockValueNet)
BlockQNet(bp, control_sym=...)               # Q(I, c)     — over iset + control
```

Alternative: a single `BlockValueNet(bp, moment="pre"|"post"|"action", ...)`
with the input dimension and required symbols derived from `moment`.

Either way, the API should:

1. **Make the moment explicit** in the class/constructor signature — no more
   "value network" that could mean either `V_pre` or `V_post`.
2. **Derive input symbols from the moment**, not by ad-hoc choice of
   `get_arrival_states()` vs `iset`.
3. **Provide a built-in `E_η` operator** for the `V_post → V_pre` integration
   (today this would be implemented per-algorithm; centralizing it removes a
   class of bugs).
4. **Key off `control_sym`** for everything that has an agent-specific iset, so
   multi-agent extends naturally — agent `i` gets its own
   `BlockDecisionValueNet` / `BlockQNet` over `I_i`.
5. **Stop returning `set`** from `Block.get_arrival_states()` — return an
   ordered sequence so multi-arrival-state models have deterministic column
   ordering across actor, critic, and replay buffer.

### Migration path

1. (i130 in-scope) Fix `BlockQNet` to use `iset`. Update `DDPG.train()`
   accordingly. This is the minimum to make DDPG correct on D-3 / U-1.
2. (Follow-up issue) Introduce `BlockArrivalValueNet` for `V_pre`. Audit
   existing algorithms (`vbi`, `maliar`) and migrate any implicit-`V_pre` code
   to use it.
3. (Follow-up issue) Rename `BlockValueNet` → `BlockDecisionValueNet` (or add
   `moment=` param) so the distinction is visible. Provide deprecation alias.
4. (Follow-up issue) Replace `set`-returning `get_arrival_states()` with an
   ordered sequence. Audit callers.
5. (When multi-agent lands) Verify that `BlockDecisionValueNet` and `BlockQNet`
   over agent-specific isets work without further refactor; if not, the API was
   wrong and step 1–3 need revisiting before multi-agent ships.

### Why this matters beyond i130

- **Partial observability:** `Q(I, c)` is the only formulation that generalizes
  when the agent can't observe the full arrival state. Locking it in now makes
  POMDP work additive rather than a refactor.
- **Multi-agent:** each agent has its own info-set; there is no single "global
  arrival state" at decision time. Agent-specific `Q_i(I_i, c_i)` is the natural
  primitive, and it falls out of the iset-based API for free.
- **Code clarity for collaborators:** today, "which value function does this
  network represent?" requires reading the constructor. With explicit moments in
  the API, it's visible from the type.
